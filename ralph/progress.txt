## Codebase Patterns
- Road rendering is in `renderRoad(ctx, scrollOffset)` — rumble strips and lane dashes both use scrollOffset modulo their period to compute starting y position
- To scroll visuals top-to-bottom (toward player), use `y = offset - PERIOD` (not `y = -offset`)
- Game states are objects with `onEnter()`, `onExit()`, `update(dt)`, `render(ctx)` methods managed by `fsm`
- gameOverState.render() is fully independent of renderRoad() — no scroll-related regression risk when modifying road rendering
- Opponents spawn at `y = -type.height` (above screen) and move down via `v.y += visualSpeed * dt`
- Speed lines already scroll correctly (positive offset direction via modulo wrap)
- FEEDBACK_WEBHOOK_URL constant was pre-added to main.js (needed for US-006)
- Opponent visual speed formula: `visualSpeed = getEffectiveScrollSpeed() * (1 - speedRatio)` — always positive, computed dynamically each frame (not stored at spawn)
- Game state variables (fsm, gameOverState, etc.) are module-scoped, NOT accessible from Playwright evaluate() — browser testing limited to visual screenshots
- Avoid long `sleep` between Playwright actions — browser context closes; navigate+act+screenshot must be done in rapid sequence
- `renderRankingPanel(ctx, rankingData, rankingStatus, rankingScroll, rankingDotTime, highlightName)` is now a shared function at line ~4486 — use it for any new state that needs a ranking panel
- HTML overlay buttons (#ranking-btn, #ranking-back-btn) are created once at startup, positioned via positionRankingBtn()/positionRankingBackBtn() called in resizeCanvas() — show/hide in each state's onEnter()/onExit()
- `titleRankingAbortController` aborts pending fetches when titleRankingState.onEnter() is called again — prevents stale data on repeat opens
- Declare HTML element vars as `null` BEFORE the initial resizeCanvas() call so position guard functions don't crash; create the actual elements AFTER
