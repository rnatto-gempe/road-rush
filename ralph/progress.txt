## Codebase Patterns
- The game is ~5900 lines in a single file: `src/main.js`
- Module-level `let` variables (e.g. `spawnPauseTimer`, `comboMultiplier`) hold state that `resetGameState()` must reset
- `resetGameState()` resets ALL game state — always add new state resets there
- Easing/utility functions go near top of file after constants section
- Phase transition update goes in `playingState.update()` after the score update block
- Build: `node_modules/.bin/vite build` (vite not in PATH; use local binary)
- Pre-generated data arrays (speedLineData, skyCloudData, spaceStarData) go near top of file after constants, use IIFE pattern
- Phase-specific rendering uses extracted helper functions: `renderSkyElements()`, `renderSpaceElements()`, `renderRoadRumbleStrips()`, `renderRoadSurface()` — reused by both static and transition renderers
- `PHASE_BG_COLORS` maps phase names to `{top, bottom}` hex colors for gradient interpolation
- `lerpColor(hex1, hex2, t)` utility near top of file for RGB interpolation between hex colors
- Transition rendering uses `ctx.globalAlpha` to fade elements in/out — always save/restore `globalAlpha`
- `renderPlayer()` uses early-return per phase — add new phases as `if (phase === 'X' || transitioning to/from X) { ... return; }` before default road car code. Order matters: space block before sky block (sky→space transition has `from === 'sky'`)
- `chooseVehicleType()` filters by `t.phase === gameState.phase` — new vehicle types MUST have a `phase` property to appear in the correct phase
- When a phase has only aggressive vehicle types (no passive), `chooseVehicleType` falls back to all available types to avoid empty pool crash
- Vehicle sprites are rendered in `renderTraffic()` with `else if (v.type === 'typeName')` chain — add new types there
