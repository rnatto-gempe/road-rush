## Codebase Patterns
- Offscreen canvas for effects: initialize at module level (not lazily) with `document.createElement('canvas')` — avoids per-frame allocation overhead
- For chromatic aberration with canvas 2D: snapshot via `drawImage(canvas, 0, 0)`, then extract color channel using `globalCompositeOperation = 'multiply'` + solid color fill, then composite back with `'screen'` at globalAlpha
- Effect state variables follow pattern: `fooTimer` (countdown), `fooDuration` (initial duration for t=timer/duration ratio), `fooIntensity` (peak scale)
- When an effect timer spans state transitions (e.g., explosion entry → explosion state), decrement the timer in BOTH `playingState.update()` and `explosionState.update()`, and render in both `render()` functions
- gameState is module-scoped (not on window) — browser evaluate can't access it; use screenshots or debug mode (D key) to verify game state
- ctx.createRadialGradient(cx, cy, r1, cx, cy, r2) is the pattern for vignette overlays — both fuel and speed vignettes use this
- All visual overlays (vignettes, flashes) use fillRect(0, 0, CANVAS_WIDTH, CANVAS_HEIGHT) after setting fillStyle to the gradient
- Render order in playingState.render(): road → [motion ghost] → traffic/items → player → particles → [capture ghost] → overlays → HUD (draw HUD last so overlays don't darken it)
- Motion blur ghost: draw offscreen canvas AFTER renderRoad() (so background is fresh) but BEFORE renderTraffic(); capture AFTER renderParticles() but BEFORE overlays — keeps HUD and overlays ghost-free
- The game runs at 60fps fixed timestep (FIXED_DT = 1/60); scrollSpeed ramps from 200 to 800 px/s over the run
- Bloom pattern: clear offscreen canvas, set `filter = 'blur(8px)'`, draw shapes at 2× scale, reset `filter = 'none'`, composite onto main canvas with `globalCompositeOperation = 'lighter'`; must call AFTER captureGhostFrame
- Particle type flags: add typed flags (e.g., `isNitro: true`) at spawn time rather than comparing color strings in the render loop
- Perspective trapezoid road: only draw geometry changes (fovTopLeft/fovTopRight at y=0, ROAD_LEFT/ROAD_RIGHT at y=CANVAS_HEIGHT); all game logic keeps using original constants. Converging lane lines use `xAtY(y) = laneXTop + (laneXBottom-laneXTop)*(y/CANVAS_HEIGHT)` extended beyond canvas for scrollable dashes
- Camera roll pattern: `ctx.save()` → `ctx.translate(CW/2, CH/2)` → `ctx.rotate(roll*PI/180)` → `ctx.translate(-CW/2,-CH/2)` → draw world → `ctx.restore()` → draw HUD; post-process effects (chroma snapshot) go after restore
- Per-band horizontal distortion pattern: draw full source canvas undistorted first, then overdraw target region with per-band `drawImage` slices each shifted by `sinOffset` — avoids edge gap artifacts; 4px band height balances quality vs draw calls
- Beat detection for visuals: queue scheduled AudioContext beat times in `AudioManager.beat.pendingBeats[]`, consume each frame via `currentTime >= pendingBeats[0]` — precise, frame-rate independent, zero polling cost
- Exponential decay for pulse animations: precompute `Math.pow(0.001, FIXED_DT / durationSecs)` as a per-frame multiplier; threshold at 0.001 before zeroing; gives snappy feel vs linear decay
- Floating pickup text pattern: single `floatTexts[]` array `{x,y,timer,text,color}` + `spawnFloatText(x,y,text,color)` helper with 8-max cap; tick in playingState.update(); render after world ctx.restore() but before HUD so texts are above world and unaffected by camera roll

- Game logic (player/traffic positions, ROAD_LEFT/ROAD_RIGHT/LANE_WIDTH) completely unaffected
- Files changed: src/main.js
- **Learnings for future iterations:**
  - Perspective trapezoid: only the DRAW geometry changes — all game logic (collision, spawn, player position) continues to use ROAD_LEFT/ROAD_RIGHT constants
  - "Extend beyond canvas" dash scrolling technique works for diagonal/converging lines too: compute `xAtY(y) = laneXTop + dx*(y/CANVAS_HEIGHT)` then draw from `(xAtY(-dashScrollOffset), -dashScrollOffset)` to `(xAtY(H+DASH_PERIOD), H+DASH_PERIOD)` — no lineDashOffset needed
  - roadCenter = 200 is the axis of symmetry for all perspective calculations; fovFactor shrinks each side equally
  - At 200 px/s (initial speed), fovFactor ≈ 5% — very subtle convergence; visually "no change" matches the AC requirement
